## Zenoh User Meeting 2025 — Recap

**ZUM25** took place on **December 12, 2025** at ZettaScale headquarters in Paris, France. It brought together practitioners from robotics, maritime systems, fleet management, and security research — all building on Zenoh.

### Highlights from the talks

**Robotics and fleet management** dominated the morning sessions. Several teams described how they moved from DDS or MQTT to Zenoh for robot-to-robot coordination, noting the elimination of discovery overhead as a key operational benefit in dynamic fleet environments.

**Maritime systems** featured a presentation on using Zenoh for ship-to-shore communication. The team leveraged Zenoh's ability to operate peer-to-peer in harbor (when connectivity is reliable) and switch to a store-and-forward model on open water — all with the same application code and no configuration changes.

**Secure group communications** was the afternoon's technical highlight. A research team demonstrated Zenoh carrying encrypted group messages with per-group key management, using queryables for key distribution. The result: end-to-end encrypted pub/sub with no trusted broker required.

### Zenoh 2.0 announced

Angelo Corsaro announced **Zenoh 2.0**, targeting **H2 2026**. The major changes:

- **Zenoh-ID routing** — routing based on stable session identifiers rather than locators, enabling seamless mobility.
- **New session handshake** — simplified and more robust, with better support for multi-homed endpoints.
- **Simplified codebase** — lessons from `zenoh-nostd` applied to the core, removing assumptions that required heap allocation.
- **Configuration refactoring** — a unified, documented configuration schema replacing the current ad-hoc structure.

Full details will be published as the implementation progresses.

## Technology Highlight: Physical AI

The January issue takes a step back from protocol engineering to look at **Physical AI** — the emerging field of AI systems that act in the physical world. Zenoh is increasingly foundational infrastructure for these systems, so understanding the field matters.

Three schools of thought dominate the discourse:

### Computational Intelligence (the mainstream view)

The dominant paradigm: intelligence emerges from computation — from large models trained on large data. Physical AI, in this view, is about scaling the same transformer architectures that work for language and vision to embodied agents. **Boston Dynamics**, **Figure**, and **1X** broadly represent this approach.

### Physical Intelligence (Meitin Sitti's view)

[Meitin Sitti](https://www.is.mpg.de/person/msitti) (Max Planck Institute for Intelligent Systems) argues that intelligence in physical systems cannot be separated from the body. His work on soft robotics and microrobotics shows that morphology — the shape and material properties of the body — is itself a computational resource. A soft gripper that conforms to objects "computes" a grasp without any algorithm.

This has implications for how we think about communication: in a physically-intelligent system, the robot's body and its environment share information through direct physical coupling, supplementing what is transmitted over a network. Zenoh's low-latency, high-frequency pub/sub fits naturally here — the network becomes an extension of physical coupling.

### Embodied Intelligence (Rodney Brooks's view)

[Rodney Brooks](https://rodneybrooks.com) — creator of the Roomba and co-founder of iRobot — has long argued (since his 1990 "Elephants Don't Play Chess" paper) that intelligence is inseparable from a body acting in the real world. He is skeptical of large models as a path to general physical intelligence, preferring reactive, behavior-based architectures grounded in sensorimotor loops.

From a Zenoh perspective: Brooks's reactive architectures depend on **extremely low-latency feedback loops**. Zenoh's sub-millisecond local latency and shared memory transport are direct enablers.

### Why this matters for Zenoh

Regardless of which school prevails, Physical AI systems share requirements:

- High-frequency, low-latency sensor data streams
- Distributed coordination across robot fleets
- Resilience to intermittent connectivity
- The ability to scale from a microcontroller (motor driver) to a data-center (training pipeline)

Zenoh addresses all of these. As Physical AI moves from research to deployment, Zenoh is positioned to be the communication substrate.

## Release Tracker: Zenoh Jiāolóng

The January release carries the codename **Jiāolóng** (蛟龙) — the Chinese aquatic dragon, associated with rivers, seas, and rain. A fitting name for a release focused on flow control and co-localization.

### Query Cancellation

A long-requested feature: **cancellation tokens for queries**. Previously, a query that timed out or was superseded by a new query could not be cancelled — it would run to completion, consuming network and compute resources.

With Jiāolóng:

```rust
let cancellation_token = CancellationToken::new();
let token_clone = cancellation_token.clone();

// Start a long-running query
let query_future = session.get("sensors/**")
    .with_cancellation(token_clone)
    .await?;

// Cancel it from another task
cancellation_token.cancel();
```

The cancellation signal propagates to all queryables serving the query, allowing them to stop processing early. This is particularly valuable for real-time systems where a stale result is worse than no result.

### Zenoh-Pico Co-localization

**Co-localization** in Zenoh-Pico eliminates serialization overhead when a publisher and subscriber run in the same process (or on the same microcontroller).

Previously, even for in-process communication, Zenoh-Pico would serialize data to bytes, pass it through the Zenoh routing layer, and deserialize it on the other side. Co-localization detects when publisher and subscriber are co-located and delivers a direct pointer to the original data structure — zero copies, zero serialization.

For microcontrollers running multiple Zenoh-Pico tasks (e.g., a sensor driver publishing to a local filter that re-publishes processed data), this eliminates a significant source of latency and CPU overhead.

## New Project: zenoh-fs

**`zenoh-fs`** is a new project building a **latency-tolerant distributed filesystem** on top of Zenoh's storage and query subsystem.

### Motivation

Traditional distributed filesystems (NFS, HDFS, Ceph) assume reliable, low-latency network connectivity. They break or perform poorly under:

- Intermittent connectivity (field robots, maritime, aerospace)
- High-latency links (satellite, long-distance WAN)
- Asymmetric bandwidth (download fast, upload slow)

`zenoh-fs` is designed from the ground up for these conditions.

### Architecture

Files are exposed as Zenoh key expressions:

```
/fs/node-id/path/to/file
```

Reads and writes are expressed as Zenoh gets and puts. The storage layer handles:

- **Checkpointing** — writes are journaled and can be resumed after disconnection.
- **Fragment-level resumption** — large file transfers track which fragments have been received; reconnection resumes from the last confirmed fragment, not the beginning.
- **Conflict resolution** — concurrent writes from multiple nodes are resolved using vector clocks, with user-configurable merge policies.

### Current status

`zenoh-fs` is in early development (pre-alpha). The team is looking for collaborators from domains with latency-tolerant storage requirements.
