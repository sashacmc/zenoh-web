## Technology Highlight: Security by Design

It is now well understood that **60–70% of security-related issues stem from memory mismanagement**. With the increasing dependence of our society, critical infrastructures, and individual lives on digital technology, security is a growing concern. Governmental organisations such as [CISA](https://www.cisa.gov/resources-tools/resources/product-security-bad-practices) (the US cyber defense agency) have launched a *Security by Design* campaign to drive the transition toward memory-safe programming languages. As part of this campaign, companies are requested to have a defined strategy — by January 2026 — on how to move toward memory safety.

The good news is that **Zenoh is Security by Design ready**.

Zenoh is entirely written in Rust and has built-in support for:
- **Authentication** — mutual TLS, token-based
- **Encryption** — TLS and DTLS transports
- **Access control** — fine-grained key-expression-based policies

ZettaScale also provides extensions for **zero-trust** deployments for those that require it.

### The Census security assessment

[Census](https://censys.io) recently conducted a security assessment of Zenoh, and the results were extremely positive. All security concerns considered were addressed with a single noted exception: **no support for end-to-end encryption** — a consequence of hop-to-hop routing. That said, if you have no trust in your infrastructure, end-to-end encryption is relatively straightforward to implement above the protocol layer. We have already done the heavy lifting, and we are considering adding it as a `zenoh-ext` in the near future.

### What about Zenoh-Pico?

Zenoh-Pico — the Zenoh protocol implementation for microcontrollers — is currently written in C. If your microcontrollers do not communicate directly with the outside but instead route through a Zenoh router (written in Rust), you are well protected. If your microcontrollers are exposed directly, the reassurances are:

1. Zenoh-Pico undergoes **static analysis** as part of its CI pipeline.
2. It is developed primarily in **MISRA-C**, which constrains the potential for memory issues.

That said, we are preparing the transition toward Rust for microcontrollers as well. As a consequence of increasing user community requests, we are developing a **Rust `no_std` implementation** of the Zenoh protocol. We already have something basic running in our labs — stay tuned for updates in future issues.

## Release Tracker

### Zenoh 1.5.0 — The Hong (Red Dragon) Epoch

Starting with version 1.5.0, Zenoh has entered the **Hong** epoch — named for the Red Dragon. Epochs typically last around six months.

This was the **first Zenoh release to break 10 million messages per second**. Key features:

- **Faster and safer Shared Memory** — significant performance improvements and safety hardening
- **Simpler config** for access control, downsampling, and QoS — more ergonomic, less boilerplate
- **DSCP link control** — fine-tune service classes at the network layer
- **Weighted routing graphs** — express topological preferences in multi-router deployments
- **QUIC datagram support** — unreliable transport over QUIC for low-latency use cases
- **Zenoh-Pico for ThreadX** — the Azure RTOS THREADX is now a supported target

### Latest Release: 1.5.1

The main features introduced in 1.5.1 are:

1. **Transparent shared memory for large messages** — when a payload exceeds a configurable threshold, Zenoh automatically uses shared memory transport, with no application-level changes required.
2. **Downsampling for `put` and `delete` messages** — previously downsampling applied only to `push` messages; 1.5.1 extends this to `put` and `delete`, enabling correct behavior when distributed storages are in the system (you want to downsample publications but not deletions).

#### Throughput benchmarks (localhost)

| Payload size | Throughput |
|---|---|
| 8 bytes | ~11 million msgs/sec |
| 16 bytes | ~1.2 Gbps |
| 64 KB | ~410 Gbps |

For full details, see the [1.5.1 release notes](https://github.com/eclipse-zenoh/zenoh/releases/tag/1.5.1).

## Community Spotlight: Pico ROS

This month's spotlight is on **[Pico ROS](https://github.com/ubiquityrobotics/pico_ros)** from [Ubiquity Robotics](https://ubiquityrobotics.com).

Pico ROS builds a thin layer above Zenoh-Pico to bring ROS 2 abstractions to microcontrollers. It runs on any platform already supported by Zenoh-Pico, including ESP32, Arduino, and STM32 families.

The key insight: by sitting above Zenoh-Pico rather than above the ROS 2 RMW, Pico ROS avoids the overhead of the full ROS 2 stack while remaining interoperable with it. A microcontroller running Pico ROS can publish to and subscribe from a full ROS 2 system on the same network without any bridge.

A [webinar introducing and demonstrating Pico ROS](https://www.youtube.com/channel/UCslbiyiqgOAPMjCrPWIfQ5Q) is available on the Ubiquity Robotics channel.

## New Projects

### ROS-Z

Security concerns also apply to robotics. If your ROS 2 robot uses the Zenoh RMW, you are already well covered: the default ROS 2 deployment routes through a Zenoh router (written in Rust), minimising the attack surface to a single communicating entity.

That said, we have also been working on a full Rust stack called **ROS-Z**, which runs natively on Zenoh and interoperates with ROS 2. The motivations:

1. **Full Rust stack** — memory safety end-to-end, from application to wire
2. **Ergonomic Rust API** — idiomatic Rust while keeping ROS 2 abstractions (nodes, topics, services, actions)
3. **Native shared memory and zero-copy** — without going through the RMW abstraction layer
4. **Serialisation independence** — not tied to CDR; choose your own serialisation format

ROS-Z is still in early development, but we are approaching feature parity with ROS 2. Give it a try and let us know what you think.
