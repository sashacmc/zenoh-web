## The Genesis of Zenoh

*By Angelo Corsaro, creator of Zenoh*

---

The story of Zenoh begins not with a blank page, but with frustration — the productive kind that comes from seeing clearly what is broken and being unable to find the right tool to fix it.

I was serving as CTO of **PrismTech**, a middleware company, and simultaneously as co-chair of the **OMG Data Distribution Service (DDS) specification group**. My work took me deep into some of the world's most demanding distributed systems: military communications, aerospace telemetry, smart-city infrastructure. I was living at the intersection of theory and production, and the gap was uncomfortable.

### The problem with existing protocols

The late 2010s offered a reasonable menu of distributed middleware protocols, each with genuine strengths. The problem was what each of them could *not* do.

**DDS** was my home territory. I had spent years designing and implementing DDS systems, and I believed — still believe — that it got the fundamental abstraction right: **data-centric pub/sub with location transparency**. Subscribers express interest in data by topic, not by knowing where the publisher lives. This is the right model. But DDS could not scale up to Internet scope (discovery was designed for LAN environments), and it could not scale down to microcontrollers (the specification and implementations assumed a capable OS and significant RAM).

**CoAP** (Constrained Application Protocol) was designed for IoT devices and constrained networks. It works. But it is fundamentally **client/server**: a device requests data from a server at a known address. There is no location transparency. You must know where the data lives. As systems grow, this becomes a configuration and operations nightmare.

**MQTT** suffered from what I came to call the **broker paradox**. Two devices sitting on the same local network, perhaps centimeters apart, would have to route their communication through a cloud broker somewhere across the Internet. The broker was simultaneously a single point of failure, a latency multiplier, and an operational dependency. And yet MQTT thrived — because simplicity wins, and a broker at least gives you a place to put configuration.

### The missing piece: data at rest

Every protocol I could find addressed *data in motion* — the flow of messages from publisher to subscriber. The world of distributed databases addressed *data at rest* — storage and retrieval. But no protocol unified them.

The gap I kept hitting was **location transparency for data at rest**. In a DDS system, you know where your publishers are (or you don't have to care, because DDS handles it). But if you want to query a geo-distributed database — sensor readings stored across dozens of edge nodes — you have to know the address of each storage node, construct queries to each, federate the results yourself. There was no protocol that let you say "give me all temperature readings from the Pacific region" and handled routing that query to the right storage nodes transparently.

The landscape was what I started calling the **"Digital Frankenstein"** era: systems stitched together from DDS for internal pub/sub, REST APIs for storage queries, MQTT for IoT devices, and custom bridges everywhere. Each bridge was a source of latency, failure, and operational complexity.

### The vision

I wanted to design something that addressed all of this with a single coherent abstraction. The requirements were clear:

1. **One protocol** from the bare-metal microcontroller to the data-center. No bridges, no topology constraints. A sensor on a microcontroller and a query running in a Kubernetes cluster should speak the same protocol natively.

2. **Unified abstractions** for three kinds of distributed operations:
   - *Data in motion* — pub/sub, time-series streaming
   - *Data at rest* — geo-distributed queries, storage federation
   - *Computations* — queryables that compute results on demand (a distributed map-reduce, essentially)

3. **Flexible topology** — peer-to-peer for local networks, brokered for cloud-scale, or any hybrid. The topology should be a runtime configuration choice, not an architectural commitment.

4. **Zero unnecessary overhead** at every layer — wire format, discovery, routing. For constrained devices, every byte and every CPU cycle matters.

This became Zenoh.

### The name

*Zenoh* carries two layers of meaning, both deliberate.

**Philosophically**, the name is a tribute to two Zenos:

- **Zeno of Elea** (c. 490–430 BC) — the pre-Socratic philosopher famous for his paradoxes of motion and infinity. Zeno's paradoxes force you to think carefully about continuity, limits, and the relationship between discrete steps and continuous processes. Distributed systems are full of analogous problems: consistency vs. availability, synchrony vs. asynchrony, discrete messages vs. continuous streams.

- **Zeno of Citium** (c. 334–262 BC) — the founder of Stoicism. Stoic philosophy emphasizes minimalism, discipline, and the distinction between what you can and cannot control. These are virtues in protocol design: minimize what you prescribe, be disciplined about abstractions, separate mechanism from policy.

**As an acronym**: **Z**ero **E**ndpoint **N**etwork **O**verhead **H**andover. This captures the operational goal: when data moves between endpoints — whether those endpoints are microcontrollers, servers, or cloud instances — the network overhead introduced by the protocol itself should be zero. Every byte on the wire should be payload or essential protocol state, never waste.

## Zenoh 1.8.0 — Regionalization

The February release introduces **Regionalization** — a new routing architecture for multi-region deployments.

### The problem

As Zenoh deployments grow to span multiple geographic regions, the flat routing model becomes a scalability bottleneck. A publisher in Tokyo and a subscriber in São Paulo do not need to involve routers in Frankfurt — but without region-awareness, the routing topology may do exactly that.

### The gateway model

Zenoh 1.8.0 introduces **region gateways**: designated routers that aggregate traffic crossing region boundaries.

Within a region, Zenoh routes efficiently using its existing peer-to-peer and mesh topology. At region boundaries, gateway routers handle cross-region traffic, applying:

- **Traffic shaping** — aggregate multiple streams crossing the boundary
- **Priority filtering** — only forward data that has subscribers in the destination region
- **Compression** — optional payload compression for high-bandwidth cross-region streams

### Configuration

Regions are declared in the Zenoh configuration:

```json5
{
  "scouting": {
    "multicast": { "enabled": true }
  },
  "routing": {
    "router": {
      "region": "eu-west",
      "gateway": {
        "enabled": true,
        "peers": ["tcp/us-east-gateway:7447", "tcp/ap-east-gateway:7447"]
      }
    }
  }
}
```

Applications need no changes — regionalization is transparent to publishers and subscribers.

## Zenoh 2.0 — Architecture Preview

With ZUM25's announcement of Zenoh 2.0 for H2 2026, we can share more details on the architectural changes.

### Zenoh-ID routing

Today, Zenoh routing is based on **locators** — network addresses (IP:port pairs). This works well when endpoints are stable, but creates problems for mobile endpoints: a robot moving between WiFi networks changes its IP address, requiring session re-establishment and disrupting active subscriptions.

Zenoh 2.0 introduces **Zenoh-ID routing**: routing based on stable, globally-unique Zenoh session identifiers rather than network locators. The routing layer tracks which locator currently corresponds to each Zenoh-ID and updates routing tables transparently when an endpoint moves. From the application's perspective, a subscription to a mobile robot's sensor stream continues uninterrupted as the robot roams.

### New session handshake

The current Zenoh session handshake was designed before several lessons were learned from production deployments. The new handshake:

- Is explicitly versioned, enabling backward compatibility
- Supports multi-homed endpoints (an endpoint with multiple network interfaces)
- Separates authentication (who are you?) from authorization (what can you do?)
- Is simpler — reducing the surface area for bugs and edge cases

### Lessons from zenoh-nostd

The `zenoh-nostd` project (see Issue 1) revealed assumptions in the Zenoh core that require heap allocation unnecessarily. Zenoh 2.0 will incorporate these findings, making heap allocation optional in the core routing logic. This benefits not just embedded targets but also server deployments where predictable memory usage matters.

### Configuration refactoring

The current Zenoh configuration is a JSON5 document that has grown organically. Zenoh 2.0 will ship with:

- A formal configuration schema (JSON Schema)
- Complete documentation for every configuration key
- Validation at startup with actionable error messages
- Migration tooling for upgrading 1.x configurations

## New Feature: Queryable Key Expression Constraints

Zenoh 1.8.0 also introduces **queryable key expression constraints** via the `ReplyKeyExpr::Any` API.

### Background

When a queryable receives a query for a key expression (e.g., `sensors/**`), it typically replies with data under the same or a sub-expression (e.g., `sensors/room1/temperature`). Previously, the Zenoh routing layer would reject replies whose key expression did not match the original query expression.

This was too restrictive for some use cases. A queryable that aggregates data might want to reply with a key like `aggregated/sensors/room1` — clearly related, but not a sub-expression of `sensors/**`.

### The new API

`ReplyKeyExpr::Any` relaxes the constraint:

```rust
let queryable = session
    .declare_queryable("sensors/**")
    .with_reply_key_expr(ReplyKeyExpr::Any)
    .await?;

queryable.recv_async().await?.reply(
    Ok(Sample::new("aggregated/sensors/room1", data))
).await?;
```

This enables richer aggregation and transformation patterns in queryable handlers.
