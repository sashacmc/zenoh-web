## Technology Highlight: Embedded Rust

One of the most exciting frontiers for Zenoh is embedded systems — and Rust is the language making it possible.

### Why Rust for embedded?

Rust offers **memory safety without a garbage collector**, making it uniquely suited for bare-metal execution where you control every byte. Unlike C, Rust's ownership model catches entire classes of bugs at compile time — dangling pointers, use-after-free, data races — all without runtime overhead.

### The embedded Rust ecosystem

The ecosystem has matured rapidly:

- **RTIC** (Real-Time Interrupt-driven Concurrency) — a framework for building real-time systems with fine-grained task scheduling and zero-cost abstractions.
- **Zephyr RTOS** — now has first-class Rust support, letting you write Zephyr applications in Rust with access to the full Zephyr API.
- **FreeRTOS** — the Rust `freertos-rust` crate allows writing FreeRTOS tasks and synchronization primitives in safe Rust.
- **ESP-HAL 1.0** — Espressif's official Rust hardware abstraction layer for ESP32 chips hit 1.0, marking production readiness for one of the most popular WiFi microcontroller families.

Zenoh's Rust implementation is well-positioned to run on all of these platforms, with `zenoh-pico` (the C implementation) covering the remaining corners.

## Release Tracker

### Shared Memory — Typed Buffers and Allocators

The Zenoh shared memory subsystem received a major API overhaul: typed buffers and pluggable allocators. Previously, shared memory was a flat byte region; now you can express typed allocations (e.g., a buffer of `f32` sensor readings) and swap allocators (POSIX, custom) without changing application code.

### Full SHM API in Python, C, and TypeScript

The shared memory API — previously Rust-only — is now available in:

- **Python** (`zenoh` PyPI package) — `ShmProvider`, `ShmBuffer`, allocator hooks
- **C** (`zenoh-c`) — full parity with the Rust API via C bindings
- **TypeScript** (`zenoh-ts`) — browser and Node.js, using WebAssembly shared memory

### Zenoh-Pico TLS

`zenoh-pico`, the lightweight C implementation targeting microcontrollers, now supports **TLS** transport. This means IoT devices with constrained resources can now participate in encrypted Zenoh sessions without a gateway.

### NuZe — Nushell Scripting for Zenoh

**NuZe** is a new Nushell plugin that exposes Zenoh operations as shell commands. You can now write Zenoh workflows as structured shell scripts:

```nushell
zenoh pub "demo/hello" --payload "world"
zenoh sub "demo/**" | each { |msg| $msg.payload | from json }
zenoh get "demo/sensors/**" | where value > 42
```

## New Project: zenoh-nostd

**`zenoh-nostd`** is a new community project implementing Zenoh pub/sub for Rust `no_std` / `no_alloc` environments, targeting the [Embassy](https://embassy.dev) async executor.

This is significant: Embassy is the leading async runtime for embedded Rust, and `zenoh-nostd` lets you write:

```rust
#[embassy_executor::task]
async fn zenoh_task(session: Session) {
    let publisher = session.declare_publisher("sensors/temp").await.unwrap();
    loop {
        let temp = read_temperature();
        publisher.put(temp).await.unwrap();
        Timer::after_secs(1).await;
    }
}
```

...without a heap allocator, on hardware like the STM32 or nRF52 series.

## Deployment Tip: Zenoh on WiFi Routers

A community member shared a workflow for running a Zenoh router on consumer WiFi routers running **OpenWRT**.

### Why routers?

- Always-on, low-power network infrastructure
- Every device on the local network has a natural Zenoh rendezvous point
- Eliminates the need for cloud broker round-trips for local pub/sub

### How-to

1. Cross-compile `zenohd` for the router's architecture (typically MIPS or ARM). The Zenoh CI already publishes `mips-unknown-linux-musl` and `armv7-unknown-linux-musleabihf` binaries.
2. SCP the binary to the router: `scp zenohd root@192.168.1.1:/usr/bin/`
3. Create an OpenWRT init script to start Zenoh at boot.
4. Configure devices on the network to connect to `tcp/192.168.1.1:7447`.

**Result:** communication overhead was halved compared to routing via a cloud broker, with sub-millisecond local latency between devices.

### MoonBots — Zenoh vs DDS for Lunar Exploration

The MoonBots team published a comparison of Zenoh and DDS for lunar surface exploration robots. Key findings:

- Zenoh achieved **3× lower latency** on the constrained lunar communication links modeled in simulation.
- DDS struggled with broker discovery overhead in high-latency, intermittently-connected scenarios.
- Zenoh's peer-to-peer mode eliminated single points of failure critical for autonomous operation far from Earth.
